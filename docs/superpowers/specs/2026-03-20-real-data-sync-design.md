# 旧生产真实菜品数据同步到重构版设计

## 目标

- 以旧生产站 `dish + dishspec` 为真实来源，把商家已维护的数据同步到重构版。
- 不迁套餐模板，不迁历史菜单，不直接改线上公开 API。
- 今天先实现可重复执行的导出、dry-run 和本地导入能力。

## 已确认规则

- 同步范围只包含 `dish + dishspec`。
- 旧生产 `110` 道菜在新版库中全部同名命中；其余新版独有菜保持不变。
- `9` 道时价菜按旧生产空值覆盖，不沿用新版占位价。
- `11` 道多规格菜按旧生产 `price_text` 重建新版规格，例如 `半只/只`、`半打/打`、`中份/大份`。
- `min_price` 不作为商家真实数据来源，但今天字段先不删，导入时按兼容规则用同步后的 `cost` 重算。
- 上线方式先走新地址验收，再决定切换旧入口。

## 实现轮廓

- 新增 `backend/scripts/sync_real_dishes.py`
  - `export`：登录旧生产并导出快照 JSON。
  - `sync`：读取快照，输出 dry-run 报告；加 `--apply` 才写入本地库。
- 对 `99` 道单规格菜保留新版规格结构，只更新价格/成本数据。
- 对 `11` 道多规格菜删除现有规格并按旧生产重建，默认规格取与 dish 主价格一致的档位。
- 写库前自动备份本地 SQLite，保证可回滚。

## 验证

- dry-run 必须输出：命中数、缺失名、ID 不一致、时价空值名单、多规格重建名单。
- 导入后抽检普通菜、时价菜、多规格菜各至少 1 个样本。
- 前端需验证菜品管理、菜单编辑器、套餐创建菜单三条主链路。
