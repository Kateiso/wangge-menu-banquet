# Part A: 交付版（2月16-18日）

## A1. 菜品管理增强 — 招牌菜/必点菜标签

### 目标
在已有的菜品管理界面上增加「招牌菜」「必点菜」标签，让 AI 生成菜单时优先选取。

### 后端变更

**`backend/models/dish.py`** — 新增字段:
```python
is_signature: bool = False    # 招牌菜
is_must_order: bool = False   # 必点菜
```

**`backend/routers/dish.py`** — DishUpdate schema 新增:
```python
is_signature: Optional[bool] = None
is_must_order: Optional[bool] = None
```

**`backend/services/menu_engine.py`** — `build_prompt()` 修改:
- 查询所有 `is_signature=True` 或 `is_must_order=True` 的菜品
- 在 Prompt 中注入:
```
## 招牌菜/必点菜约束
以下菜品为本店招牌或必点，请至少选择 3 道：
- #12 旺阁即烧烧鹅（招牌）
- #24 豉油皇乳鸽（必点）
...
```
- 招牌菜数量从偏好中读取（如果用户选了"要招牌菜"），否则默认 min(3, 招牌菜总数)

### 前端变更

**`frontend/src/components/DishManager.tsx`** — Table columns 新增:
- 「招牌菜」列: Switch 开关
- 「必点菜」列: Switch 开关
- 编辑弹窗中新增这两个字段

**`frontend/src/api/menuApi.ts`** — Dish 类型新增 `is_signature`, `is_must_order`

### 关键文件

| 文件 | 操作 |
|------|------|
| `backend/models/dish.py` | 修改: +2 字段 |
| `backend/routers/dish.py` | 修改: DishUpdate +2 字段 |
| `backend/services/menu_engine.py` | 修改: build_prompt 注入招牌菜 |
| `frontend/src/components/DishManager.tsx` | 修改: +2 列 + Switch |
| `frontend/src/api/menuApi.ts` | 修改: Dish 类型 +2 字段 |

## A2. 按位菜品数量逻辑优化

### 目标
解决"14人点14只乳鸽"的问题。让 AI 理解按位菜品的合理数量。

### 后端变更

**`backend/models/dish.py`** — 新增字段:
```python
serving_unit: str = ""        # "例" | "只" | "件" | "斤" | "份"（从 price_text 自动解析）
serving_split: int = 0        # 一开几 (0=不适用/不确定, 2=一开二, 4=一开四)
```

**`backend/services/dish_service.py`** — `parse_price()` 增强:
- 从 price_text 自动提取 serving_unit（如 "99元/只" → serving_unit="只"）
- serving_split 暂时默认 0，等老板确认后手动在菜品管理中设置

**`backend/services/menu_engine.py`** — `build_prompt()` 修改:
- 原来第 69 行: `按只/件计价的菜（乳鸽、T骨等），数量按人数配`
- 改为: 遍历有 serving_split > 0 的菜品，在 Prompt 中列出具体规则:
```
## 按位菜品数量规则
- 按只/件计价的菜品，不要按人数1:1配置
- 已设定规则的菜品:
  - #24 豉油皇乳鸽: 一开二，{party_size}人建议 {ceil(party_size/2)} 只
- 未设定规则的按位菜品: 按人数的 40-60% 配置数量
```

### 前端变更

**`frontend/src/components/DishManager.tsx`**:
- 编辑弹窗新增:「上菜单位」(Input) + 「一开几」(Select: 不适用/一开二/一开四)
- Table 中有 serving_split > 0 的菜品显示小标签 "一开二" 等

### 关键文件

| 文件 | 操作 |
|------|------|
| `backend/models/dish.py` | 修改: +2 字段 |
| `backend/services/dish_service.py` | 修改: parse_price 提取 unit |
| `backend/services/menu_engine.py` | 修改: build_prompt 数量规则 |
| `frontend/src/components/DishManager.tsx` | 修改: 编辑弹窗 +2 字段 |

## A3. 毛利率/预算控制精度提升

### 目标
减少毛利率偏差（从 ±4% → ±3%）和预算超支。

### 后端变更

**`backend/services/menu_engine.py`** — `build_prompt()`:

1. **预算约束加强** (第 65-68 行):
   - 当前: "允许范围 {budget_low}~{budget_high}"
   - 改为更强的措辞 + 验算要求:
   ```
   3. 预算控制【最重要，必须遵守】:
      - 目标总价: {budget} 元，绝对不能超过 {budget_high} 元
      - 最低消费: {budget_low} 元，不能低于此金额
      - 选完所有菜后，请逐一计算 price×quantity 求和，确认总价在范围内
      - 如果超预算，删掉最贵的一道菜或减少数量
   ```

2. **毛利率约束加强** (第 70 行):
   ```
   4. 毛利控制【重要】:
      - 目标整单毛利率 {target_margin}%，允许偏差 ±3%
      - 每道菜都有标注成本，请计算: 毛利率 = (总售价-总成本)/总售价×100%
      - 如果毛利偏低，优先替换低毛利菜品为同类高毛利菜品
   ```

3. **validate_and_build_menu()** 增强:
   - 超预算 > 1.05x 时，自动触发重试（当前只 log warning）
   - 毛利偏差 > 5% 时，自动触发重试

**`backend/services/menu_engine.py`** — `generate_menu()`:
- 当前: 只在预算利用率 < 75% 时重试
- 增加: 预算超支 > 105% 或毛利偏差 > 5% 也触发重试
- 重试时 Prompt 中加入具体偏差说明

### 关键文件

| 文件 | 操作 |
|------|------|
| `backend/services/menu_engine.py` | 修改: prompt 措辞 + 验证逻辑 + 重试条件 |

## 数据库迁移策略

由于项目使用 `删除 menu.db 重建` 方式（无迁移），Dish 表新增字段后需要:

1. 删除 `menu.db`
2. 重启后端（自动重建表 + CSV 导入）
3. 新字段（is_signature, is_must_order, serving_unit, serving_split）初始值均为默认值
4. 老板通过菜品管理界面手动标记招牌菜和设置一开几

# Part B: 进化版（交付后，不在本次实施范围）

> 以下为交付后的长期规划，仅做记录，不在2月18日前实施。
> 详细的 Agentic Agent 技术调研见同目录下的 `Agentic_Agent_技术调研.md`

## B1. 规则基础设施
- 新建 SystemRule 表：把所有硬编码参数（成本率、品类分布、预算容差等）搬到数据库
- 新建 RuleService：规则 CRUD + 格式化为 Prompt 文本
- 修改 menu_engine.py 和 dish_service.py 从规则读取参数
- 为 Agentic 自优化打基础

## B2. 反馈引擎（Agentic 核心）
- 新建 FeedbackEngine 服务
- 老板自然语言反馈 → LLM 解析为规则变更建议 → 老板确认后写入 SystemRule → 下次生成自动生效
- 流程: 反馈 → 分析 → 建议 → 确认 → 应用
- API: POST /api/feedback, POST /api/feedback/{id}/apply

## B3. 后验评估器
- 每次生成菜单后自动评分（预算利用率、毛利偏差、品类覆盖、招牌菜数量）
- 累积评估数据，发现系统性偏差
- 生成参数微调建议（需老板确认才生效）

## B4. 多租户架构
- 数据隔离: 所有表新增 tenant_id
- 每家餐厅独立: 菜品库、规则库、用户体系
- 部署方案: 共享实例（低成本）vs 独立实例（高隔离）

## B5. 前端规则管理 UI
- 规则列表: 查看、启用/停用、手动编辑
- 反馈历史: 状态追踪（待处理/已应用/已拒绝）
- 评估报告: 可视化图表
